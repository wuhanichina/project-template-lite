# ProjectName 对外叙事参考

> 本文件用于面向小同行、合作者和工程人员解释项目。正文推荐使用
> “起、承、转、合、升”的宏观逻辑，从工程实际问题出发，逐步揭示关键关系、
> 物理机理和现有不足，再说明本文处理方式与有范围的工程含义。创新层级和
> 技术对象决定各环节的叙事角度、篇幅重心和解释深度。

## 1. 确定工程入口和叙事重心

主创新轴决定文章讲述的中心，技术对象决定需要解释的关系，叙事路线决定
“起、承、转、合、升”各环节的篇幅和重心。文章默认从可核查的工程对象、
工况、现象或决策问题进入，不从方法名称、内部代号或抽象概念进入。

- `research-object`：先说明过去没有被单独识别、计算或评价的工程对象，
  再说明识别该对象后哪些问题可以被提出和验证。
- `discovery-mechanism`：先呈现可复核的现象、失配、反例或边界，再建立
  变量之间的作用关系和可验证预测。
- `technical-object`：先说明现有模型或计算流程在哪个对象、约束或条件下
  失效，再解释本文技术对象改变了哪一层关系。
- `engineering-decision-loop`：先呈现规划、运行、调度、定价、控制或风险
  评估中的具体选择，再说明新增信息如何改变判断依据。

创新幅度只改变论证重点，不改变事实。`zero-to-one` 需要突出新对象、新现象
或新解释及其边界；`one-to-hundred` 应突出已知对象上的模型精度、计算条件、
适用范围或决策改进，不把性能改进改写成新物理机理。

## 2. 在推荐逻辑中配置叙事重心

各路线都保留“起、承、转、合、升”的总体推进：工程问题是“起”，研究对象
和现有处理是“承”，关键关系、物理机理与不足是“转”，本文处理和证据是
“合”，有范围的工程含义是“升”。填写时选择一条主路线，用它决定哪个环节
展开、哪个环节压缩。`mixed` 也应确定一个主要重心，避免平均分配篇幅。

### 路线 A：发现优先

适用于 `discovery-first`。

1. 从工程系统中的一个有来源运行现象、失配、反例或边界条件开始，明确系统和工况。
2. 说明常见解释为什么不能覆盖该现象，并给出已排除因素的证据。
3. 展开关键变量、状态或约束之间的关系，给出能够被推导或算例检验的预测。
4. 说明该发现改变了对研究对象的哪项认识，以及它对后续模型或工程判断的作用。

正文起笔：

【待补充：在什么系统和工况下，哪个可观察量出现了什么现象，证据来自哪里。】

### 路线 B：机理优先

适用于 `mechanism-first`。

1. 从工程对象的状态变化或约束冲突开始，不从方法名称开始。
2. 沿功率、能量、电压、电流、频率、概率或时空过程说明作用链。
3. 指出作用链在哪个环节形成现有方法无法解释的差异。
4. 用独立参照、推导或边界算例检验该作用关系，而不是只报告综合指标。

正文起笔：

【待补充：哪个物理过程通过哪些状态量和约束产生了当前工程现象。】

### 路线 C：方法优先

适用于 `method-first`。

1. 先说明工程任务及其对输入、输出、方程、约束或计算能力的要求。
2. 定位现有模型、算法或控制流程的具体缺口。
3. 解释本文方法改变的模型映射、计算性质、状态响应或成立条件。
4. 依次给出正确性、可信参照、最近基线和适用边界证据。

正文起笔：

【待补充：现有计算在哪个对象、条件或约束下留下什么可验证缺口。】

### 路线 D：工程决策优先

适用于 `engineering-first`。

1. 从工程人员必须作出的一个具体选择开始，说明可选动作和错误代价。
2. 说明当前决策缺少哪个状态量、风险量、边界或评价依据。
3. 解释本文输出如何由物理状态或模型结果转化为决策信息。
4. 用匹配工况说明决策发生了什么变化，同时保留成本、风险和适用范围。

正文起笔：

【待补充：谁在什么条件下需要作出什么选择，当前信息缺口影响什么结果。】

### 路线 E：验证优先

适用于 `validation-first`。

1. 先说明工程上需要确认的现象或判断，再用真实测量、可信参考解、标准算例或独立实验构造验证问题。
2. 说明参考对象为什么足以检验本文关注的关系。
3. 展示本文结果与参照的一致部分、残差结构和失配条件。
4. 在完成可信性论证后，再讨论方法比较和工程用途。

正文起笔：

【待补充：哪一项可信参照能够区分本文方法与现有解释，它覆盖什么范围。】

### 路线 F：边界诊断优先

适用于 `boundary-diagnosis`。

1. 从工程运行或计算中出现的方法失效、结果反转、误差突增或约束失守开始。
2. 区分数据不足、模型遗漏、参数范围、数值求解和物理条件造成的差异。
3. 给出边界形成的可检验解释，以及边界内外结果如何变化。
4. 将贡献落在适用条件识别、模型修正或工程风险规避上。

正文起笔：

【待补充：哪个条件触发了什么失配，现有证据能够把原因定位到哪一层。】

## 3. 解释到正确的层级

“为什么有效”不自动等于“发现了新物理机理”。正文按技术对象解释对应关系：

- `model-formulation`：解释工程对象如何映射为变量、方程和约束，新增或修正
  的模型关系是什么。
- `algorithm-solution`：解释问题的分解、近似、误差、收敛或复杂度来源，以及
  算法利用了什么计算性质。
- `control-strategy`：解释状态、观测、动作和响应之间的闭环关系，以及扰动和
  安全边界。
- `theory-guarantee`：解释结论成立所需的假设、可解性、唯一性、误差界或
  稳定条件及其工程含义。
- `data-case-system`：解释数据覆盖、测量可信度、可观察对象和验证能力，
  不把数据规模本身写成物理发现。

只有当推导或独立证据建立了物理变量之间的因果链及其可验证预测时，才把
项目判断写成物理机理。其他贡献应准确写成模型关系、计算性质、状态响应、
成立条件、观测能力或决策关系。

## 4. 形成学长或工程专家的口吻

成熟的工程叙事依靠判断顺序，而不是修辞堆叠：

1. 先交代系统对象、工况和可观察事实。
2. 再指出最容易产生的误判，以及该判断忽略了哪个量或条件。
3. 沿一条关系链解释结果，不同时展开多条互不相干的技术支线。
4. 用一个关键图、公式、数值或反例完成验证。
5. 说明工程人员因此可以改变哪项判断，同时明确不能外推到哪里。

正文应使用连续段落。表格、字段名、claimId 和 evidenceId 留在定位卡或文末
证据说明中，不让审计字段打断叙事。

结尾使用有范围的工程判断：

> 在【待补充：适用范围】内，【待补充：研究对象、模型关系、计算性质或
> 决策关系】为【待补充：工程任务】提供【待补充：可验证输出】。该判断由
> 【待补充：证据来源】支持，当前证据状态为【尚未验证】。

## 6. 与论文和汇报材料的关系

- 对外文章可以使用选定的叙事路线，但必须保留范围、证据来源和证据状态。
- Introduction 和 Contributions 必须从 `ProjectName_note.md`、`01_IDEA/`、
  登记结果和已核查文献重新组织，不能轻量改写本文件后直接使用。
- 答辩和项目汇报可以压缩叙事路线，但不能把模型关系、计算性质或验证能力
  统一拔高为新物理机理。
- `story.md`、`claims.md`、`evidence_map.md`、研究结果或文献核查状态变化
  时，应重新检查定位卡、正文主线和文末证据说明。
