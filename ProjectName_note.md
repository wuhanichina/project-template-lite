## 0. 写作定位与证据状态

### 写作边界

- 不按论文的“引言-方法-算例-结论”压缩表达，而是记录问题形成、路线演化、方案取舍和证据状态。
- 不隐藏失败结果、被放弃路线、不成熟模型、未完成推导或当前局限。
- 不使用“突破性”“革命性”“彻底解决”“全面优于”“首次实现”“完全避免”等强表述，除非已经有充分文献检索、理论证明和定量实验支撑。
- 若信息缺失，使用 `【待补充】`、`【尚未验证】`、`【文献待核查】`、`【当前证据不足】`、`【Git 待提交】`、`【文档滞后】`、`【尚未实现】`、`【代码待核查】`、`【结果待复核】`，不要补写成事实。

### 证据状态标签

对每个判断性结论标注证据状态：

| 状态 | 使用场景 |
| --- | --- |
| 已由理论推导支持 | 有清晰公式、假设和推导链，但不一定已有数值验证 |
| 已由数值实验支持 | 有登记过的 case、结果文件、指标和配置 |
| 已由代码实现支持 | 有可定位的函数、入口或测试，但结果可能尚未充分验证 |
| 已有 Git 版本记录 | 有 branch、commit、tag、issue 或 PR 可追溯 |
| 仅为当前假设 | 目前用于推进建模，但尚无充分证据 |
| 尚未验证 | 需要补推导、补实验、补代码或补文献 |
| 结果不稳定 | 不同参数、样本、算例或重复运行下表现不一致 |
| 当前证据不足 | 已有材料不能支撑当前强度的表述 |
| 已被实验排除 | 已有实验表明该判断不成立 |
| 暂时放弃 | 当前不进入主线，但保留复盘价值 |
| 文献待核查 | 需要补最近竞争者或理论出处 |
| 代码尚未实现 | 文档有方案，代码还没有 |
| 结果待复核 | 有输出，但尚未完成一致性和质量检查 |

### 5.5 基本模型

先写最小可验证模型：

$$
\min_{x,u,z} \quad F(x,u,z;\theta)
$$

subject to

$$
G_i(x,u,z;\theta) \le 0,\quad i=1,\ldots,m,
$$

$$
H_j(x,u,z;\theta) = 0,\quad j=1,\ldots,p.
$$

其中：

- $F(\cdot)$ 表示【待补充】。
- $G_i(\cdot)$ 表示【待补充】。
- $H_j(\cdot)$ 表示【待补充】。

若使用松弛、近似、线性化、矩匹配、场景法、分解或 SCA，应明确原始问题、变换后问题、等价或近似条件，以及近似误差如何影响最终 claim。

### 5.6 求解策略
